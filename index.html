OGC API - Records
Bringing data to the users
Tell Me More Get Started

Ensure your data is not buried in the deep web

It is estimated that more than 80% of the web content is not indexed by search engines. This includes content that is intentionally hidden such as private repositories, but also some organisation specific repositories, academic databases and government reports.

Do you create geospatial resources, like data, services or ML models? OGC API – Records gives you Standard interfaces to expose your resources and a Standard vocabulary to describe them.

This description will make sure that search engines will index the resources in a structured way, so that humans and machines can find them, understand their descriptions and fetch them.

Why?

Main reasons to implement a catalogue using OGC API - Records

Interoperable

The Standard is developed and maintained by an independent Standards body, which also offers compliance and certification, ensuring that interoperability really works. In addition, there was an harmonisation with the STAC specification , in order to make sure that STAC catalogues can understand OGC API - Records catalogues and vice versa.

Flexible

OGC API - Records supports different deployment patterns, which address different use cases. For instance, a public directory with satellite imagery can benefit from having a crawlable catalogue (e.g.: also called “static” catalogue), which can be created with static files without the need to install any middleware or databases.

Based on Best Practices

The Standard is based on the technologies and vocabularies that make up the web today, such as REST, OpenAPI, JSON and HTML. This should ensure a smooth learning curve for any web developer, even if they are not familiar with GIS.

How does it work?

The OGC API - Records Standard provides discovery and access to metadata about geospatial resources. Once a metadata record is found, it provides a way of fetching the associated data.

The Standard offers three building blocks:

  • A record
  • A collection of records
  • A records API
When these three building blocks are combined together they enable developers to implement a searchable catalog , a powerful way of publishing metadata.

For use cases where search is not essential, there is the alternative of deploying a crawlable catalog , which only requires some files deployed along the data in a web accessible location. This deployment pattern imposes a very low implementation burden because it relies on HTTP to do most of the work of accessing the records of a catalog.
